\section{Εισαγωγή}

\subsection{Σκοπός}
Ο σκοπός αυτού του εγγράφου είναι να παρέχει μια σαφή και αναλυτική επισκόπηση της αρχιτεκτονικής σχεδίασης και των διαδικασιών της πλατφόρμας “CoachFlow”.

\subsection{Έκταση}
Το έγγραφο καλύπτει την υλοποιημένη αρχιτεκτονική του CoachFlow, διαχωρισμό React Frontend και PHP REST API, session-based authentication με CSRF, ροές αγοράς/ανάθεσης προγραμμάτων, διατροφή με USDA integration, real-time messaging με Firebase Firestore, και υποσυστήματα διαχείρισης (Trainer/Admin).

\subsection{Ορισμοί}
\textbf{React SPA (Single Page Application)} \\
Το Frontend σύστημα που τρέχει αποκλειστικά στον Web Browser του χρήστη και φορτώνει τα components χωρίς να χρειάζεται να ανανεωθεί ολόκληρη η σελίδα.

\textbf{RESTful API} \\
Η αρχιτεκτονική που χρησιμοποιεί το Backend (PHP) για να εκθέτει endpoints (\textenglish{URLs}) τα οποία καταναλώνονται από το Frontend μέσω \textenglish{GET, POST, PUT, DELETE requests}.

\textbf{JSON} \\
Η μορφή μεταφοράς δεδομένων (payloads) ανάμεσα στο React Web App και τον PHP Web Server μας. 

\textbf{Session-based Authentication} \\
Η βασική υλοποίηση εξουσιοδότησης στο backend γίνεται μέσω PHP sessions (cookie με \textenglish{credentials: include}) και όχι μέσω bearer JWT για τα business endpoints.

\textbf{Firestore Messaging} \\
Το chat υλοποιείται σε Firebase Firestore (conversations/messages) με custom token που εκδίδεται από το backend μετά από επιτυχημένο session authorization.

\subsection{Program Assignment Flow}
Η πλατφόρμα υποστηρίζει δύο πρώτης κλάσης ροές για τα προγράμματα:
\begin{itemize}[label=•]
    \item \textbf{Marketplace Flow:} Trainer δημοσιεύει πρόγραμμα (\textenglish{trainer\_programs.status = published}), Trainee αγοράζει μέσω Stripe, και αποκτά πρόσβαση μετά από επιβεβαίωση πληρωμής.
    \item \textbf{Direct Assignment Flow:} Trainer αναθέτει υπάρχον πρόγραμμα απευθείας σε Trainee (program reuse), χωρίς υποχρεωτικό βήμα αγοράς.
\end{itemize}

Το παρακάτω διάγραμμα δείχνει τη ροή της \textbf{direct assignment}. Η ροή του marketplace συνδέεται με τη διάθεση του προγράμματος και την πληρωμή μέσω Stripe, και για αυτό καλύπτεται και από το επόμενο payment flow. Δεν είναι το διάγραμμα της αυθεντικοποίησης που εμφανίζεται πιο πάνω. Η MySQL δεν επικοινωνεί απευθείας με τον trainer· επιστρέφει το αποτέλεσμα στο backend, και το backend ενημερώνει το frontend.

\subsection{Messaging Architecture (Firestore)}
Το chat είναι real-time και βασίζεται σε Firestore. Ο χρήστης λαμβάνει custom Firebase token από το backend endpoint \textenglish{GetFirebaseCustomToken.php}, και στη συνέχεια η frontend εφαρμογή συνδέεται σε conversation documents/messages με rules based authorization.

\subsection{Ρόλοι Συστήματος}
Οι βασικοί ρόλοι είναι Guest, Trainee, Trainer, Admin. Ο ρόλος \textenglish{manager} αντιμετωπίζεται λειτουργικά ως παραλλαγή admin με περιορισμένα δικαιώματα, χωρίς ξεχωριστό ανεξάρτητο υποσύστημα στο παρόν design document.

\section{Ασφάλεια Συστήματος}

\subsection{Session \/ Role-based Authorization}
Το backend επιτρέπει πρόσβαση μόνο σε authenticated sessions. Οι protected endpoints ελέγχουν role privileges (trainee, trainer, admin, manager) πριν την επιχειρησιακή λογική.

\subsection{CSRF Protection}
Το σύστημα δημιουργεί CSRF token στο session και το συγχρονίζει με cookie (\textenglish{XSRF-TOKEN}). Τα state-changing requests ελέγχονται με CSRF header για αποτροπή cross-site request forgery.

\subsection{CORS και Cookie Policy}
Χρησιμοποιείται ελεγχόμενο CORS policy για επιτρεπόμενα origins, ενώ τα session cookies είναι \textenglish{HttpOnly} και \textenglish{SameSite} για καλύτερη προστασία.

\subsection{Password / Recovery Security}
Οι κωδικοί αποθηκεύονται hashed στη MySQL. Τα flows email verification και reset password χρησιμοποιούν time-bound tokens.

\subsection{Payment Security}
Η τιμή αγοράς δεν εμπιστεύεται frontend payload. Το backend διαβάζει τιμή από DB και δημιουργεί PaymentIntent. Επιπλέον, τα webhook events του Stripe ελέγχονται με signature verification πριν ενημερωθεί η κατάσταση αγοράς.

\section{Deployment Σχεδίαση (Cloud-agnostic)}

\subsection{Τοπολογία Υψηλού Επιπέδου}
\begin{itemize}[label=•]
    \item Frontend SPA served μέσω static hosting/CDN.
    \item PHP API σε application server layer.
    \item Managed MySQL-compatible database.
    \item Εξωτερικές υπηρεσίες: Stripe, Firebase, USDA, Mailchimp.
\end{itemize}

\subsection{Περιβαλλοντικές Μεταβλητές και Secrets}
Όλα τα API keys και credentials (DB, Stripe, Firebase service account, Mailchimp, SMTP) διατηρούνται σε environment variables και δεν εκτίθενται στο frontend bundle.

\subsection{Backup / Recovery (Brief)}
Προτείνεται τακτικό DB backup policy και test restore διαδικασίες για λειτουργική συνέχεια. Το design document κρατά συνοπτική αναφορά χωρίς επιχειρησιακές SLA λεπτομέρειες.

\section{Συμπέρασμα}
Η σχεδίαση του CoachFlow υλοποιεί πλήρως decoupled web αρχιτεκτονική με σαφή separation concerns μεταξύ Frontend, Backend και Database. Υποστηρίζει ολοκληρωμένο lifecycle για Trainee/Trainer/Admin (και manager ως admin variant), ασφαλείς πληρωμές, real-time messaging, και επεκτάσιμες ροές προγραμμάτων (marketplace + direct assignment), καλύπτοντας τις βασικές ανάγκες της πλατφόρμας.
